第 2 周学习计划（Sanctum 登录与受保护接口）

目标：在 7 天内走通「注册/登录/登出 + 受保护接口 + React 鉴权状态」最小闭环。
每天：3~5 小时（调试、查错、记录都算有效学习时间）

目录

  • 第 1 天：认证模型梳理与基线接口
  • 第 2 天：注册接口 + 基础校验
  • 第 3 天：登录接口 + Token 签发
  • 第 4 天：受保护接口与未授权响应
  • 第 5 天：React 接入登录并保存凭证
  • 第 6 天：React 访问受保护接口 + 401/403 处理
  • 第 7 天：登出、回归测试与周复盘
建议：每天下线前记录「今天跑通了什么、还卡在哪里、明天第一步做什么」。

第 1 天：认证模型梳理与基线接口

1, 目标
明确本周采用的认证模式（Personal Access Token）；确认已有后端接口可用，并建立统一 JSON 返回格式基线。
2, 学习内容
Sanctum 在 API Token 模式下的核心流程：注册/登录签发 Token，后续请求走 Authorization: Bearer xxx。
3, 参考资料
Laravel Sanctum
Laravel Authentication
4, 练习
  1. 在当前后端项目新增文档 `week2-auth-notes.md`，写下「注册、登录、受保护接口、登出」四步流程。
  2. 确认基础探活接口可访问（例如你第 1 周的 `/api/hello`）。
  3. 约定返回结构（示例）：成功 `{ "ok": true, "data": ... }`，失败 `{ "ok": false, "message": "..." }`。
完成标准
你能用自己的话解释 Token 流程，并有一份本周认证流程笔记。

第 2 天：注册接口 + 基础校验

1, 目标
实现注册接口：能创建用户，字段校验失败时返回 422 且错误结构清晰。
2, 学习内容
Request 校验、密码加密、唯一性校验；理解 422 与 400 的常见语义差异。
3, 参考资料
Laravel Validation
HTTP 422 · MDN
4, 练习
  1. 创建注册接口（示例路径：POST /api/register）。
  2. 至少验证：name 必填、email 格式且唯一、password 最小长度。
  3. 使用 Postman 或 curl 测试 2 组数据：一组成功、一组故意失败并记录响应体。
完成标准
注册成功能拿到用户信息；非法输入返回 422 且错误字段可读。

第 3 天：登录接口 + Token 签发

1, 目标
实现登录接口：邮箱密码正确时签发 Token；错误时返回统一未授权结构。
2, 学习内容
密码校验、Token 生成与命名（便于后续多端管理）；HTTP 401 语义。
3, 参考资料
Issuing API Tokens
HTTP 401 · MDN
4, 练习
  1. 实现 POST /api/login：正确凭证返回 token，错误凭证返回 401。
  2. 把 token 存在本地测试文件中（只用于本地，不提交 Git）。
  3. 用 curl 带上 Authorization: Bearer TOKEN 调试请求，确认 token 可被识别。
完成标准
登录成功可拿到 token；错误密码稳定返回 401。

第 4 天：受保护接口与未授权响应

1, 目标
新增受保护接口并挂上 auth:sanctum，验证「带 token 成功，不带 token 401」。
2, 学习内容
中间件保护路由、请求用户上下文、基础权限边界（已登录 vs 未登录）。
3, 参考资料
Protecting Routes
4, 练习
  1. 新增接口（示例：GET /api/me），返回当前用户基础信息。
  2. 先不带 token 请求，确认是 401。
  3. 再带 token 请求，确认返回用户数据。
  4. 把两次结果（状态码 + 响应体）记录到 `week2-auth-notes.md`。
完成标准
受保护接口行为符合预期：未登录拒绝，已登录放行。

第 5 天：React 接入登录并保存凭证

1, 目标
前端完成最小登录页：提交账号密码，拿到 token，并在请求层自动携带。
2, 学习内容
受控表单、异步请求状态、TypeScript 接口类型；token 存储策略（仅本地开发可先简化）。
3, 参考资料
React 状态管理基础
Axios 请求配置（若你项目使用 axios）
4, 练习
  1. 新增登录表单（email/password）。
  2. 请求成功后保存 token（如 `localStorage`），并在请求封装里自动带 Bearer 头。
  3. 登录失败时在页面显示错误文案，不要只看控制台。
完成标准
前端可完成一次真实登录，且能在浏览器存储中看到 token。

第 6 天：React 访问受保护接口 + 401/403 处理

1, 目标
前端可调用受保护接口并展示用户信息；凭证失效时能统一处理并回到登录态。
2, 学习内容
请求拦截、错误分流（401 与 403）、鉴权状态管理最小设计。
3, 参考资料
HTTP 403 · MDN
条件渲染 · React
4, 练习
  1. 页面加载时请求 /api/me 并渲染用户名。
  2. 手动删除 token 再刷新，确认触发 401 逻辑并回到登录提示。
  3. 补一个最小错误页或错误提示组件（至少包含状态码与重试按钮）。
完成标准
登录态与未登录态切换清晰，401/403 不会“静默失败”。

第 7 天：登出、回归测试与周复盘

1, 目标
完成后端登出（撤销 token）与前端退出登录，做一轮小回归并输出复盘。
2, 学习内容
接口闭环思维：注册 → 登录 → 访问受保护接口 → 登出 → 再次访问失败。
3, 参考资料
Revoking Tokens
4, 练习
  1. 实现 POST /api/logout，撤销当前 token。
  2. 前端点击「退出登录」后清理本地 token，并跳回登录页。
  3. 执行回归清单：注册、登录、访问、登出、再访问；记录结果。
  4. 写一段周复盘：本周最大卡点、怎么解决、下周想优化什么。
完成标准
整个鉴权闭环可重复演示，且有可追溯的测试记录与复盘文字。

  • 认证流程笔记（`week2-auth-notes.md`）
  • 注册、登录、受保护接口、登出四类接口的测试截图或文本记录
  • 前端登录态页面（含错误提示）
  • 本周复盘（至少 5 行）
